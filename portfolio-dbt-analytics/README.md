# analytics-pipeline — dbt Core Analytics Pipeline

A production-grade dbt Core analytics pipeline targeting **Amazon Redshift Serverless**, migrated from hand-written SQL ETL. Orchestrated by **AWS MWAA 2.7.2** in production and iterated locally with the **dbt Fusion engine**.

---

## Architecture

### 5-Layer Warehouse

```
analytics_source  (raw landing zone — external tables / Redshift Spectrum)
       ↓
analytics_dw      (source_data + meta_data + etl_data)
       ↓
analytics_aggregation  (aggregation_data)
       ↓
analytics_report  (report_data — BI-facing views)
```

| Layer | Schema | Materialisation | Purpose |
|---|---|---|---|
| `source_data` | `analytics_source` | incremental (append) | Unified ODS event table from multi-source union |
| `meta_data` | `analytics_metadata` | view | Dimension seeds and lookup tables |
| `etl_data` | `analytics_dw` | incremental (delete+insert) | Cleaned fact and dimension tables |
| `aggregation_data` | `analytics_aggregation` | incremental (delete+insert) | Pre-aggregated retention and funnel metrics |
| `report_data` | `analytics_report` | view | Thin BI-facing reporting views |

## Key Technical Patterns

### 1. Jinja Event Union Macro (`macros/event_macros.sql`)

All mobile app events (Nova app) land in separate Redshift tables per event type. A single macro — `analytics_event_union` — generates a `UNION ALL` across every event table and normalises them to a common schema with:

- Standard envelope columns (`event_name`, `event_guid`, `event_timestamp`, `client_*`, `product_*`)
- A JSON `payload` column built via `OBJECT()` from event-specific context columns
- Transparent handling of dev vs. production source databases

```sql
-- generated by analytics_event_union
{% for event_def in analytics_event_definitions() %}
  {{ analytics_event_union(event_def) }}
  {% if not loop.last %} UNION ALL {% endif %}
{% endfor %}
```

### 2. Incremental Models with Idempotent Delete Pre-Hook

Every incremental model uses a `pre_hook` to DELETE the target partition before insertion, making every run fully idempotent:

```sql
pre_hook = "DELETE FROM {{ this }}
            WHERE partition_date >= {{ analytics_partition_lower_bound('nova') }}
              AND partition_date <= {{ analytics_airflow_ds() }}"
```

This avoids Redshift `MERGE` cost while guaranteeing safe reruns and backfills.

### 3. Backfill Window Control via dbt Vars

The backfill window is controlled entirely through dbt variables — no code changes required for ops:

```bash
# Reprocess the last 7 days
dbt run --select source_data --vars '{"airflow_ds": "2024-03-15", "analytics_backfill_days": 7}'

# Override backfill window per event type
dbt run --vars '{"backfill_days_by_event": {"device_session_record": 14, "user_auth_record": 3}}'
```

Variables are injected by the Airflow DAG from Airflow Variables, allowing ops to trigger backfills without code deploys.

### 4. Multi-Database Targeting

All models resolve their target database and schema from dbt vars, enabling the same codebase to write to different Redshift databases for dev/staging/production:

```yaml
# dbt_project.yml
vars:
  analytics_database: analytics_dw
  analytics_schema_source: analytics_source
  analytics_schema_etl: analytics_dw
  # ...
```

```sql
-- in model config
database = var('analytics_database'),
schema   = var('analytics_schema_etl')
```

### 5. Rolling Dimension via Temp-Table Trick (`dim_user_product_relationship`)

Redshift Serverless has no native SCD Type 2 / snapshot support in dbt. This model uses a `CREATE TEMP TABLE AS SELECT * FROM this` pre-hook pattern to atomically swap today's snapshot while preserving historical rows — without a full table scan or external orchestration.

## CI/CD

```
Push to develop branch
       → GitHub Actions: dbt compile + sqlfluff lint
       → S3 sync to dev MWAA DAGs bucket
       → Dev MWAA picks up new DAG version

Publish GitHub Release (tag vX.Y.Z)
       → GitHub Actions: dbt compile + sqlfluff lint
       → S3 sync to production MWAA DAGs bucket
       → Production MWAA picks up new DAG version
```

Production deployments are gated by the GitHub Release publish event, providing a clear audit trail of what code ran in production and when.

## Local Development

```bash
# Install dependencies
pip install dbt-redshift dbt-core

# Install dbt packages
dbt deps

# Compile (no DB connection needed)
dbt compile

# Run source layer for yesterday with 3-day backfill
dbt run --select source_data \
  --vars '{"airflow_ds": "2024-03-14", "analytics_backfill_days": 3}'

# Run full pipeline
dbt run --vars '{"airflow_ds": "2024-03-14"}'

# Generate and serve docs
dbt docs generate && dbt docs serve
```

## Environment Variables / Secrets

All secrets are stored in **AWS Secrets Manager** and injected at runtime by the Airflow DAG. No credentials appear in code or dbt profiles committed to the repository.

| Secret key | Purpose |
|---|---|
| `redshift/host` | Redshift Serverless endpoint |
| `redshift/user` | DB user |
| `redshift/password` | DB password |
| `redshift/database` | Target database |
| `redshift/port` | Port (default 5439) |
